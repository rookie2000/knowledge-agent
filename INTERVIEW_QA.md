# 面试问答准备

## 1. 项目介绍

### Q: 请介绍一下你的项目

**一句话回答：**
Knowledge Agent 是一个基于 RAG + Tool Use 的智能文档助手，用户上传文档后，可以通过自然语言提问获取答案。

**展开回答：**
"这个项目的核心是让 Agent 能够理解用户问题，自动决定是否调用工具，从知识库中检索相关信息，然后生成准确的回答。与普通 Chatbot 不同，它能展示完整的推理过程，包括工具调用记录和来源引用。"

**结合项目：**
"我在这个项目中实现了完整的 Agent Loop，包括工具调用、多轮对话、失败恢复等。还做了 Prompt Injection 防护和完整的 Trace 记录。"

---

## 2. 技术选型

### Q: 为什么用 Agent 而不是普通 Workflow?

**一句话回答：**
因为用户的问题是开放的，需要 Agent 动态决定调用什么工具、怎么推理，而不是固定流程。

**展开回答：**
"如果问题是固定的，比如'提取文档摘要'，用 Workflow 就够了。但用户可能问各种问题，需要 Agent 理解意图，决定是否调用工具，调用哪个工具，怎么组合结果。这是 Agent 的核心价值。"

**结合项目：**
"在我的项目中，用户可能问'什么是 RAG'，也可能问'这个项目用了什么技术'，Agent 需要动态判断是否调用搜索工具，而不是每次都调用。"

---

### Q: Agent 和普通 Chatbot 有什么区别?

**一句话回答：**
Chatbot 主要是对话生成，Agent 能规划、调用工具、处理失败。

**展开回答：**
"Chatbot 接收输入，生成回答，是单步的。Agent 能：
1. 规划执行步骤
2. 调用外部工具
3. 获取实时信息
4. 处理失败和重试
5. 记录执行过程"

**结合项目：**
"我的项目中，Agent 不只是回答问题，还会调用搜索工具获取文档内容，引用来源，如果搜索失败还会重试或降级。"

---

### Q: 为什么选择 ChromaDB 而不是 Pinecone?

**一句话回答：**
MVP 阶段用本地部署的 ChromaDB，无需外部依赖，后续性能不够再考虑 Pinecone。

**展开回答：**
"ChromaDB 的优点是：
1. 本地部署，无需外部服务
2. Python 原生支持，集成简单
3. 足够 MVP 阶段使用

缺点是性能一般，不适合大规模数据。但如果后续需要，可以迁移到 Milvus 或 Pinecone。"

**结合项目：**
"我的项目是个人知识库，数据量不大，ChromaDB 完全够用。如果要做企业级应用，可能需要考虑更专业的方案。"

---

### Q: 为什么自研 Agent 而不用 LangChain?

**一句话回答：**
自研能完全控制逻辑，面试时能展示对底层原理的理解。

**展开回答：**
"LangChain 优点是开箱即用，但缺点是：
1. 抽象层厚，难调试
2. 性能有损耗
3. 不利于理解底层原理

自研的优点是：
1. 完全可控
2. 易于调试和优化
3. 能展示对 Agent 原理的理解"

**结合项目：**
"我选择自研，是因为这个项目的目标之一是展示 Agent 开发能力。用 LangChain 可能更快，但面试官问底层原理时，我可能答不上来。"

## 3. 核心设计

### Q: 工具调用是怎么设计的?

**一句话回答：**
Claude 返回 tool_use 时，我执行对应工具，将结果发回 Claude，循环直到返回最终回答。

**展开回答：**
"工具调用的流程是：
1. 定义工具 Schema（名称、参数、描述）
2. 调用 Claude API 时传入工具定义
3. Claude 返回 tool_use 时，执行对应工具
4. 将工具结果发回 Claude
5. 循环直到 Claude 返回最终回答

关键点是工具 Schema 要清晰，让 Claude 知道什么时候用什么工具。"

**结合项目：**
"我定义了两个工具：search_knowledge 和 get_document_info。Claude 会根据问题自动决定调用哪个。"

---

### Q: Agent 什么时候调用工具，什么时候不调用?

**一句话回答：**
如果问题涉及文档内容，调用搜索工具；如果问题与知识库无关，直接回答。

**展开回答：**
"调用工具的判断逻辑是：
1. 问题涉及文档内容 → 调用 search_knowledge
2. 询问文档信息 → 调用 get_document_info
3. 与知识库无关 → 直接回答

这个判断是 Claude 自动做的，通过工具描述来引导。"

**结合项目：**
"我在工具描述中明确写了'搜索知识库中的相关文档片段'，Claude 就知道什么时候该用这个工具。"

---

### Q: 工具调用失败怎么办?

**一句话回答：**
重试 1 次，失败后返回错误提示或降级策略。

**展开回答：**
"失败处理策略是：
1. 工具超时 → 重试 1 次
2. 返回空结果 → 尝试不同查询词
3. 工具异常 → 记录 Trace，返回通用错误

关键是要优雅降级，不能让整个系统崩溃。"

**结合项目：**
"我在 AGENT_DESIGN.md 中详细定义了各种失败场景和处理策略，确保 Agent 在异常情况下也能正常工作。"

---

### Q: 如何防止 Agent 产生幻觉?

**一句话回答：**
要求 Agent 只基于文档内容回答，并引用来源。

**展开回答：**
"防止幻觉的方法：
1. 要求 Agent 只基于检索到的内容回答
2. 强制引用来源，便于验证
3. 如果检索不到相关内容，说明无法回答
4. 在 eval 中测试幻觉率"

**结合项目：**
"我在 Prompt 中明确要求'只能基于知识库内容回答'，并且返回结果必须包含来源引用。"

---

## 4. 安全相关

### Q: 如何防止 Prompt Injection?

**一句话回答：**
系统指令与用户输入分离，用户输入作为 data 不作为 instruction。

**展开回答：**
"Prompt Injection 防护措施：
1. 系统指令与用户输入严格分离
2. 用户输入作为 data，不作为 instruction
3. 工具结果也作为 data 处理
4. 高风险操作需要二次确认
5. 在 eval 中加入 Prompt Injection 测试用例"

**结合项目：**
"我在 SECURITY.md 中详细定义了安全策略，并在 evals 中加入了 Prompt Injection 测试用例，确保 100% 防御率。"

---

### Q: 如何保护敏感信息?

**一句话回答：**
API Key 从环境变量读取，日志脱敏，错误信息不暴露内部细节。

**展开回答：**
"敏感信息保护措施：
1. API Key 从环境变量读取，不硬编码
2. 日志和 Trace 中脱敏处理
3. 错误信息不包含内部细节
4. 限制 Agent 只能访问知识库，不能访问系统文件"

**结合项目：**
"我在 .env.example 中定义了配置项，实际的 API Key 放在 .env 中，不会提交到代码仓库。"

## 7. 性能相关

### Q: 响应时间是多少?

**一句话回答：**
简单问题 < 2s，复杂问题 < 5s，取决于工具调用次数和 LLM 响应速度。

**展开回答：**
"响应时间取决于：
1. 工具调用次数：每次调用约 0.5-1s
2. LLM 响应速度：约 1-2s
3. 检索速度：约 0.1-0.5s

优化方向：
1. 减少工具调用次数
2. 优化 Prompt 减少 Token 用量
3. 使用缓存减少重复调用"

**结合项目：**
"我在 PERF_COST_REPORT.md 中记录了性能基准和优化过程。"

---

### Q: 成本如何控制?

**一句话回答：**
通过 Prompt 优化、缓存、减少不必要的工具调用来控制成本。

**展开回答：**
"成本控制方法：
1. Prompt 优化：减少 Token 用量
2. 缓存：相同问题不重复调用
3. 工具调用优化：避免不必要的调用
4. 模型选择：简单问题用小模型"

**结合项目：**
"我在 Prompt 中要求简洁回答，减少了 Token 用量。后续可以加入缓存机制进一步优化。"
